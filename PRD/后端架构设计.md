# 迷局AI — 后端架构设计

> **版本**: v1.0 | **日期**: 2026-03-30 | **状态**: 待审核

---

## 一、设计背景与目标

### 1.1 现状分析

迷局AI 当前是一个**纯前端应用**，存在以下架构瓶颈：

| 维度 | 当前方案 | 问题 |
|------|----------|------|
| 数据存储 | localStorage | 清除即丢失、跨设备不同步、5-10MB 限额 |
| AI 调用 | 前端直连 LLM API | API Key 暴露在客户端，安全风险极高 |
| 用户体系 | 无 | 无法追踪用户行为、无社交互动基础 |
| 社区功能 | 仅本地 | 剧本无法真正分享，社区广场需要云端支持 |
| 多人玩法 | 不支持 | 无实时通信能力 |

### 1.2 后端目标

- **安全代理 AI 调用** — API Key 服务端托管，前端零暴露
- **云端数据持久化** — 用户档案、游戏存档、自定义剧本云同步
- **社区生态基础** — 剧本发布/浏览/评分/收藏的完整后端支撑
- **可扩展架构** — 为多人联机、语音交互、成就系统预留扩展点
- **渐进式迁移** — 前端 API 调用方式平滑过渡，不破坏现有离线体验

## 二、技术选型

### 2.1 核心技术栈

| 层级 | 选型 | 理由 |
|------|------|------|
| **运行时** | Node.js 20 LTS | 与前端共享 JS 生态，降低全栈学习成本 |
| **Web 框架** | Express 5.x | 项目已引入 (package.json)，成熟轻量，中间件生态丰富 |
| **数据库** | MongoDB (Mongoose ODM) | 剧本/对话等半结构化数据天然适合文档模型，Schema 灵活 |
| **缓存** | Redis | 会话存储、AI调用限流、排行榜计数 |
| **认证** | JWT (Access + Refresh Token) | 无状态认证，Web/小程序/App 通用 |
| **AI 代理** | OpenAI SDK (兼容层) | 统一 OpenAI/DeepSeek/Gemini 等 API 的调用格式 |
| **实时通信** | Socket.IO (预留) | Phase 4 多人联机需要 |
| **日志** | Pino | 高性能结构化日志 |
| **校验** | Zod | TypeScript-first 的运行时校验（JS 中同样可用） |
| **测试** | Vitest + Supertest | 与 Vite 生态统一，API 集成测试 |

## 四、API 设计 (RESTful)

### 4.1 认证模块 `/api/auth`

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|
| POST | `/api/auth/register` | 注册（邮箱/手机号） | ❌ |
| POST | `/api/auth/login` | 登录，返回 accessToken + refreshToken | ❌ |
| POST | `/api/auth/refresh` | 刷新 accessToken | ❌ (需 refreshToken) |
| POST | `/api/auth/logout` | 登出，invalidate refreshToken | ✅ |
| POST | `/api/auth/guest` | 游客登录（设备标识） | ❌ |

> [!IMPORTANT]
> **游客模式关键设计**：为保持"打开即玩"的体验，支持游客模式。游客自动生成 `guest_xxx` 账号，数据保留 30 天。正式注册后可绑定游客数据。

### 4.2 用户模块 `/api/users`

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|
| GET | `/api/users/me` | 获取当前用户资料 | ✅ |
| PATCH | `/api/users/me` | 更新用户资料（昵称/头像） | ✅ |
| GET | `/api/users/me/stats` | 获取用户统计（游玩/创作数据） | ✅ |
| GET | `/api/users/:id/profile` | 查看他人公开主页 | ✅ |

### 4.3 剧本模块 `/api/scenarios`

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|
| GET | `/api/scenarios/presets` | 获取预设剧本列表（公开） | ❌ |
| GET | `/api/scenarios/:id` | 获取剧本详情 | ❌ (社区) / ✅ (私有) |
| POST | `/api/scenarios` | 创建自定义剧本（保存到云端） | ✅ |
| PATCH | `/api/scenarios/:id` | 编辑自定义剧本 | ✅ (所有者) |
| DELETE | `/api/scenarios/:id` | 删除自定义剧本 | ✅ (所有者) |
| GET | `/api/scenarios/my` | 获取我的剧本列表 | ✅ |

### 4.4 游戏会话模块 `/api/games`

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|
| POST | `/api/games` | 创建游戏会话 | ✅ |
| GET | `/api/games/:id` | 获取游戏状态 | ✅ |
| GET | `/api/games/saves` | 获取存档列表 | ✅ |
| POST | `/api/games/:id/chat` | 发送对话（代理 AI） | ✅ |
| POST | `/api/games/:id/deduction` | 提交推理 | ✅ |
| DELETE | `/api/games/:id` | 删除存档 | ✅ |

### 4.5 社区模块 `/api/community`

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|
| GET | `/api/community/scenarios` | 浏览社区剧本（搜索/筛选/排序/分页） | ❌ |
| POST | `/api/community/publish` | 发布剧本到社区 | ✅ |
| POST | `/api/community/:id/favorite` | 收藏/取消收藏 | ✅ |
| GET | `/api/community/favorites` | 我的收藏列表 | ✅ |
| POST | `/api/community/:id/review` | 提交评价（1~5 星 + 文字） | ✅ |
| GET | `/api/community/:id/reviews` | 获取剧本评价列表 | ❌ |
| GET | `/api/community/rankings` | 排行榜（热门/高分/最新） | ❌ |

### 4.6 AI 代理模块 `/api/ai`

| 方法 | 路径 | 说明 | 认证 |
|------|------|------|------|
| POST | `/api/ai/chat` | NPC 对话代理（流式 SSE） | ✅ |
| POST | `/api/ai/extract-clue` | 线索提取 | ✅ |
| POST | `/api/ai/score` | 语义评分 | ✅ |
| POST | `/api/ai/generate-scenario` | AI 生成剧本 | ✅ |

> [!TIP]
> **AI 代理的核心优势**：
> 1. API Key 仅存储在服务端 `.env`，前端零暴露
> 2. 服务端按用户限流，防止 API 被滥用
> 3. 可按需切换不同模型（对话用 DeepSeek、生成用 GPT-4o 等）
> 4. Prompt 模板服务端管理，版本可控

```javascript
const UserSchema = new Schema({
  // 账号信息

    scenariosCreated:  { type: Number, default: 0 },

    totalPlayTime:  { type: Number, default: 0 }, // 分钟

  // 偏好设置
  settings: {
    theme:    { type: String, default: 'auto' },
    fontSize: { type: Number, default: 16 },
    aiModel:  { type: String, default: '' },  // 用户自选模型偏好

  // Token

## 六、关键流程设计

### 6.1 AI 对话代理流程（核心）

```
前端                        后端                           LLM API
  │                          │                               │
  │  POST /api/ai/chat       │                               │
  │  { gameId, message }     │                               │
  │ ────────────────────────►│                               │
  │                          │  1. JWT 认证                   │
  │                          │  2. 根据 gameId 加载会话        │
  │                          │  3. 构建 messages 数组          │
  │                          │     (systemPrompt + history)   │
  │                          │  4. 检查回合限制                │
  │                          │                               │
  │                          │  POST /chat/completions        │
  │                          │  (stream: true)               │
  │                          │ ─────────────────────────────►│
  │                          │                               │
  │   SSE: data: chunk       │◄──── streaming chunks ────────│
  │◄─────────────────────────│                               │
  │   SSE: data: chunk       │                               │
  │◄─────────────────────────│                               │
  │   SSE: data: [DONE]      │                               │
  │◄─────────────────────────│                               │
  │                          │                               │
  │                          │  5. 保存对话到 GameSession      │
  │                          │  6. 更新回合数                  │
  │                          │  7. 异步线索提取                │
  │                          │                               │
```

            │         → answer/truthReveal 对外隐藏
            │
            └── 失败 → 提示审核不通过原因
```

## 九、前端迁移方案

> [!IMPORTANT]
> **渐进式迁移，不破坏离线体验**

### 9.1 混合模式策略

```javascript
// 前端 AI Service 改造
class AIService {
  async chat(systemPrompt, history, userMessage, onChunk) {
    if (this.backendAvailable) {
      // 走后端代理（推荐）
      return await this._chatViaBackend(gameId, userMessage, onChunk)
    } else {
      // 降级为直连（兼容离线/未部署场景）
      return await this._chatDirect(systemPrompt, history, userMessage, onChunk)
    }
  }
}
```

### 9.2 数据同步策略

- **在线时**：数据写入后端 → 本地缓存副本
- **离线时**：数据写入 localStorage → 上线后增量同步
- **冲突解决**：以服务端时间戳为准（last-write-wins）
