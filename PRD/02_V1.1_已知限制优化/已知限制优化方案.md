## 二、各限制项优化方案

### 限制 1：单人模式

**现状**：仅支持 1 名玩家独立游玩，无多人互动。

**优化方案 — 异步协作推理模式**：

| 方案 | 说明 | 复杂度 | 推荐 |
|------|------|--------|------|
| A. 好友挑战 | 同一剧本，各自独立推理，结束后比较分数 | ⭐⭐ | ✅ 推荐 |
| B. 协作模式 | 2-4 人分别负责不同 NPC 的审讯，共享线索板 | ⭐⭐⭐⭐ | 后续迭代 |
| C. 实时对抗 | 多人同时在线，抢先推理得分加成 | ⭐⭐⭐⭐⭐ | 不推荐 |

**推荐实现（方案 A）**：

```
玩家A 完成游戏 → 生成挑战链接/小程序码
    → 玩家B 扫码/点击 → 玩同一剧本
    → 双方分数对比，展示排行
```

**涉及改动**：
- 新增 `challenge` 云数据库集合（挑战 ID、参与者、各自分数）
- 结果页增加「挑战好友」按钮 → 生成分享卡片
- 新增挑战结果对比页

**关键设计**：
- 异步模式，不需要双方同时在线
- 挑战链接自带剧本 ID，点击直接进入该剧本
- 无需 WebSocket，实现成本低

### 限制 2：无后端（数据存储不可靠）

**现状**：所有数据 `localStorage` 存储，清除缓存即丢失。

**优化方案 — 分层存储架构**：

```
┌─────────────────────────────────────────┐
│  Layer 1: 本地缓存（即时读写）            │
│  wx.setStorageSync / localStorage       │
│  用途：当前游戏快照、UI 偏好              │
├─────────────────────────────────────────┤
│  Layer 2: 云数据库（持久存储）            │
│  微信云开发 / Supabase                   │
│  用途：用户数据、存档、剧本、评价         │
├─────────────────────────────────────────┤
│  Layer 3: CDN / 云存储（静态资源）        │
│  用途：预设剧本数据、分享海报             │
└─────────────────────────────────────────┘
```

**同步策略**：

| 场景 | 行为 |
|------|------|
| 对话中每轮 | 本地缓存 → 异步写入云端（防抖 3 秒） |
| 主动保存 | 立即写入云端 + 本地 |
| 页面进入 | 云端拉取最新数据 → 与本地合并（取时间戳更新的） |
| 网络断开 | 仅写本地，恢复后自动同步 |

**Web 端轻量方案（无需云开发）**：

如果暂不上小程序，可先接入 **Supabase** 免费层：
- 500MB 数据库 + 1GB 存储 + 50K 月活
- RESTful API，前端直接调用
- 匿名登录（UUID）→ 绑定微信 / GitHub 账号

### 限制 3：评分方式（关键词匹配）

**现状**：动机/手法评分基于硬编码关键词匹配，评分精度低。

```javascript
// 当前方式：关键词命中数 / 3 × 满分
const matchCount = keywords.filter(kw => text.includes(kw)).length
const score = Math.min(matchCount / 3 * maxScore, maxScore)
```

**问题**：
- 同义词表达无法识别（如"嫉妒" vs "羡慕成恨"）
- 分数颗粒度粗糙（0/10/20/30 四档跳跃）
- 精心描述和随意输入可能得相同分数

**优化方案 — AI 语义评分 + 关键词兜底**：

```
┌──────────────────────────────────┐
│  1. 先执行关键词匹配（instant）    │
│     → 得到 keyword_score          │
├──────────────────────────────────┤
│  2. 异步调用 AI 语义评分          │
│     → 得到 ai_score              │
├──────────────────────────────────┤
│  3. 取两者较高值                  │
│     final_score = max(两者)       │
│     （保证 AI 失败时有兜底分数）    │
└──────────────────────────────────┘
```

**AI 评分 Prompt 模板**：

```
你是剧本杀游戏的评分裁判。请评估玩家推理的准确度。

## 标准答案
动机: ${answer.motive}
手法: ${answer.method}

## 玩家回答
动机: ${deduction.motive}
手法: ${deduction.method}

## 评分规则
- 动机满分 30 分，手法满分 30 分
- 核心要素完全正确 → 满分
- 核心要素部分正确 → 15-25 分
- 方向正确但细节偏差 → 8-15 分
- 完全偏离 → 0-5 分

请返回 JSON：{"motive_score": N, "method_score": N, "comment": "一句话点评"}
```

**涉及改动**：
- `ai-service.js` 新增 `scoreDeduction()` 方法
- `result.js` 中调用 AI 评分，与关键词评分取 max
- AI 评分失败时静默降级为关键词评分

### 限制 4：自定义剧本质量不可控

**现状**：AI 生成的剧本可能出现：
- 凶手与 NPC 列表不匹配
- 线索逻辑断裂（线索关联错误 NPC）
- JSON 结构不完整

**优化方案 — 三层质量保障**：

#### Layer 1：结构化校验（生成后即刻执行）

```javascript
function validateScenario(scenario) {
  const errors = []
  
  // 1. 必填字段检查
  const required = ['title', 'intro', 'npcs', 'clues', 'answer', 'truthReveal']
  required.forEach(key => {
    if (!scenario[key]) errors.push(`缺少必填字段: ${key}`)
  })
  
  // 2. NPC 数量校验
  if (scenario.npcs?.length < 3) errors.push('NPC 数量不足 3 人')
  
  // 3. 凶手必须在 NPC 列表中
  if (!scenario.npcs?.find(n => n.id === scenario.answer?.suspect)) {
    errors.push('凶手 ID 不在 NPC 列表中')
  }
  
  // 4. 线索来源 NPC 必须存在
  const npcIds = scenario.npcs?.map(n => n.id) || []
  scenario.clues?.forEach(clue => {
    if (!npcIds.includes(clue.source)) {
      errors.push(`线索"${clue.title}"的来源 NPC "${clue.source}" 不存在`)
    }
  })
  
  // 5. 每个 NPC 至少有 1 条线索
  npcIds.forEach(id => {
    if (!scenario.clues?.some(c => c.source === id)) {
      errors.push(`NPC "${id}" 没有关联任何线索`)
    }
  })
  
  // 6. 关键词非空
  scenario.clues?.forEach(clue => {
    if (!clue.keywords?.length) errors.push(`线索"${clue.title}"缺少触发关键词`)
  })
  
  return { valid: errors.length === 0, errors }
}
```

#### Layer 2：AI 自检（校验失败时）

```
校验失败 → 将错误列表 + 原始剧本发回 AI
         → 要求修复具体问题
         → 最多重试 2 次
```

#### Layer 3：用户预览确认（已实现 Step 4）

在 Step 4 预览页增加质量提示：
- ✅ 所有校验通过 → 显示「✅ 剧本结构完整」
- ⚠️ 存在问题 → 高亮问题项，提供「🔄 让 AI 修复」按钮

**涉及改动**：
- `custom-scenario.js` Step 3 → Step 4 之间增加 `validateScenario()` 调用
- 新增校验结果展示UI
- 新增 AI 修复重试逻辑

### 限制 7：API Key 明文存储

**现状**：`ai-service.js:27` → `localStorage.setItem(API_KEY_STORAGE, apiKey)` 明文存储。

**优化方案**：

| 方案 | 安全等级 | 说明 | 推荐场景 |
|------|----------|------|----------|
| A. 云函数代理 | ⭐⭐⭐⭐⭐ | Key 存在服务端，前端不接触 | ✅ 小程序版（已纳入转型 PRD） |
| B. Web Crypto 加密 | ⭐⭐⭐ | 用设备唯一标识作密钥加密 Key | Web 端过渡方案 |
| C. Session 级存储 | ⭐⭐ | Key 存 sessionStorage，关标签即清除 | 临时方案 |

**Web 端推荐 — 方案 B 实现**：

```javascript
// 使用 Web Crypto API + 设备指纹作为密钥
async function encryptApiKey(apiKey) {
  const encoder = new TextEncoder()
  const data = encoder.encode(apiKey)
  
  // 用固定 passphrase + UA 作为密钥材料
  const keyMaterial = await crypto.subtle.importKey(
    'raw', encoder.encode(navigator.userAgent + 'miju-salt'),
    'PBKDF2', false, ['deriveKey']
  )
  const key = await crypto.subtle.deriveKey(
    { name: 'PBKDF2', salt: encoder.encode('miju-ai'), iterations: 100000, hash: 'SHA-256' },
    keyMaterial, { name: 'AES-GCM', length: 256 }, false, ['encrypt', 'decrypt']
  )
  
  const iv = crypto.getRandomValues(new Uint8Array(12))
  const encrypted = await crypto.subtle.encrypt({ name: 'AES-GCM', iv }, key, data)
  
  return { iv: Array.from(iv), data: Array.from(new Uint8Array(encrypted)) }
}
```

> [!CAUTION]
> 前端加密只能提升攻击门槛，不能根本性解决安全问题。真正安全的方案是 **后端代理**（方案 A）。
